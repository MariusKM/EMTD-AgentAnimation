# HeroAnimation - Pipeline Workspace

## Purpose
This directory contains everything needed to produce hero screen animations for EMTD using an **Image-to-Video (I2V)** pipeline. Source hero art is fed as input to AI video generation models (Kling 3.0 or Seedance 2.0) to produce animated character loops.

## Workspace scope

Hero animation pipeline only (named heroes — power moves + idles, bespoke per character). Skill: [`/hero-animation`](../.claude/skills/hero-animation/SKILL.md). Source: `Source/Heroes Stylized/<HeroName>.png`. Output: `Output/<HeroName>/`.

The Dragon character runs through the hero pipeline with empty P-slots (idle-only, but with hero-pipeline expressive scope). Dragon doc: [`Docs/Heroes/Dragon.md`](Docs/Heroes/Dragon.md). Source: `Source/Heroes Stylized/Dragon.png`.

## Pipeline Overview (see `Docs/Pipeline.md` for full detail)

| Stage | What Happens | User Involvement |
|-------|-------------|-----------------|
| 1. Character Review | Review + polish hero description against source art | Collaborative refinement |
| 2. Animation Concepts | Propose power + idle concepts, refine with user | User selects and approves |
| 3. Prompt Creation | Translate concept to model-specific I2V prompt | User chooses model + reviews |
| 4. Generation | Submit via Kling API or manual upload | User may submit manually |
| 5. Review & Iterate | Evaluate output, refine prompt, re-generate | Collaborative review |
| 6. Post-Processing | External green-screen keying + `compose-frames` assembly | User runs keying, agent runs composition |
| 6.5 Alignment | Auto-detect eye+foot anchors, recompose to template-aligned variants | Batch buttons in webapp, manual override per-clip |
| 6.9 Delivery | Encode VP8/WebM with alpha at 550×550 to `Output/<Hero>/Final/` | Batch "Deliver" button in webapp |
| 7. Audio (WIP) | Add SFX via ElevenLabs | TBD |

**Key rule**: The user is involved at every decision point. Do not generate without approval.

**FFLF creation timing**: The FFLF (`Output/<Hero>/<Hero>_FFLF_<n>.png`) is produced by the **user, after Stage 2 concepts are locked and Stage 3 prompts are drafted** — not at Stage 1. The user's FFLF tooling lives in the webapp and requires the concepts to exist first (concepts inform which expression / hand-state / chroma background the FFLF should bake in). Therefore the Stage 1 audit of `Output/<Hero>/<Hero>_FFLF_<n>.png` is **expected to fail for every new hero** — flag the absence to the user and continue into Stage 2 anyway. The blocker only kicks in at **Stage 4**: do not queue generation jobs until the user confirms the FFLF has been created in the webapp. Pre-Stage-4 the agent should explicitly call out "FFLF not yet created — will be produced by user after concepts are locked" rather than treating the missing file as a stop condition.

## Working With a Character

When starting work on a hero animation, follow the pipeline stages in order:

1. **Load context**: Read `Docs/Heroes/<Name>.md` + view `Source/Heroes Stylized/<Name>.png`
2. **Concepts**: Use `PromptTemplates/animation_concepts.md` to propose ideas — get user approval
3. **Prompts**: Use `PromptTemplates/hero_animation_gen.md` with the user's chosen model (Kling/Seedance)
4. **Generate**: Via Kling API skill or manual submission
5. **Review**: Watch the generated output directly and iterate on prompts
6. **Document**: Log findings to `Output/<Name>/prompts.md` and major learnings to this `CLAUDE.md`

## Known Issues & Learnings
> This section is updated as the pipeline matures. Document major findings from generation here.

### Pipeline / Tooling
- **`concepts.md` heading format must match the webapp parser — applies to BOTH power AND idle blocks** (learned 2026-04-30 on Merchant; **recurred 2026-05-06 on Diana — idle blocks only**). The webapp at [webapp/lib/concepts.ts](../webapp/lib/concepts.ts) parses concept blocks with the regex `/^#{3,4}\s+([PI])(\d+)\s*[:\-—]\s*(.+?)\s*$/` — i.e. headings must be exactly `### P1 — Title` / `### P1: Title` / `#### I2 — Title` etc., with the concept ID (`P` or `I` + digits) **immediately after** the `###`/`####` and a colon / hyphen / em-dash separator. Headings like `### Concept P1: Hearty Gun Salute` or `### Concept I1 — Breathing Baseline` (with `Concept` before the ID) silently fail to parse — the webapp UI then shows "No concepts.md found at Output/<Hero>/concepts.md" even though the file exists, OR (when only some headings are malformed) hides the malformed concepts while showing the well-formed ones. **The Diana 2026-05-06 recurrence is instructive**: the power blocks (P1–P6) parsed fine because the author followed the template's `### P1 —` form, but the **idle blocks were written as `### Concept I1 — ...`** (because the authoring instinct was to label them as "concepts" once moving past the power section), so the webapp showed all 6 powers but **none of the 3 idles** — the prompt files for I1/I2/I3 also vanished from the UI even though they existed on disk. **Authoring rule**: when creating a new hero's `concepts.md`, lead **every** concept block — power AND idle — with the bare `### P1 — Name` / `### I1 — Name` form. After writing, grep with `^#{3,4}\s+(Concept\s+)?[PI]\d+` to verify there are zero matches with the `Concept` prefix. Cross-reference an existing working hero (Architect, Princess Sweet) for canonical format. The same `P#_` / `I#_` filename prefix is also how prompt files match to concepts in [webapp/lib/scan.ts](../webapp/lib/scan.ts) — keep prompt filenames starting with the bare concept ID.

### Audio / Vocalizations
- **The `no voice no dialogue no music` hard-ban can produce near-silent output, especially on idles** (learned 2026-04-30 on Merchant I1). The Merchant I1 first-pass used the standard double-bracket ban + "faint cloth ambience / faint wicker ambience / soft natural calm ambient nose-breathing" SFX list and came back with effectively no audio. Combination of `no voice` (which the audio model interprets aggressively) + `faint`/`almost imperceptible` SFX descriptors collapses the audio output to silence. **Fix**: drop `no voice` from idle Sound fields; replace `no voice no dialogue` with the gentler `no actual words no actual speech no music`; describe SFX with concrete audible language (`a soft wicker creak`, `a soft wooden creak`, `a soft cloth shift`) rather than "faint" / "almost imperceptible" / "ambience"; add an explicit positive Constraint `the audio includes audible ambient cloth and wicker SFX throughout the loop the audio is not silent`. Apply on every hero's I1, I2, I3 prompt by default.
- **Wordless vocalizations are a clean way to give a hero audio personality without triggering dialogue** (learned 2026-04-30 on Merchant). When a character should not be mute (Merchant is jovial — silent merchant felt wrong), invite **wordless** vocalizations explicitly in Sound: `a warm hearty welcoming wordless chuckle from the merchant on the body turn`, `a jolly hearty wordless laugh from the merchant at the moment of fire`, `a soft warm contented wordless closed-mouth hum`, `a small contented wordless grunt`. Pair with a positive Constraint: `no actual words no actual speech the merchant's vocal sounds are wordless hearty chuckles and jolly laughs and contented hums only no dialogue`. The `no actual words` framing keeps the dialogue suppression that `no voice` was doing while letting non-verbal character vocalizations through. **Authoring rule**: every hero should have a documented "vocal direction" in their `concepts.md` — what wordless vocalizations fit their personality (hearty chuckles for Merchant, dry chuckles for General, sneaky stifled laughs for Spy, regal sighs and small clicks for Fat King, sweet light gasps for Princess Sweet, etc.). Apply across power AND idle prompts; even I1 baselines get a single subtle wordless touch.
- **Both-fields invite + ban pattern** — the same redundancy lesson that applies to `no voice no dialogue no music` (Spy 2026-04-16) applies inverted to invitations: name the wordless vocalization both in the **Sound** field at the timecode it occurs AND in the **Constraints** with a positive permission clause (`the merchant's vocal sounds are wordless ... only`). Both required for reliable audio adherence.

### Tone & Style
- **Tone is jovial/slightly comical (client direction, 2026-04-16)**: All hero animations should lean jovial and slightly comical in typical Supercell fashion (Clash Royale, Clash of Clans). Every character should carry a small comedic or warm beat — a smirk, wink, flourish, shrug — even imposing ones (General, Wilhelm keep a dry smirk; Spy, Fat King, Fat Princess lean hard into it). See `Docs/StyleGuide_Animation.md` §3 "Overall Tone" for how this applies across character archetypes.
- **Princess-class heroes are POWERFUL FIRST, princess second — comedy comes from confidence, not haughtiness** (learned 2026-04-30 on Princess Sweet, recurred 2026-05-06 on Princess Arrogant). The default authoring instinct on a princess character is to lean cute / haughty / dismissive / theatrical-comedy — and on first-pass concept slates this consistently produces feedback "she doesn't feel powerful enough, she's a HERO." Princess Sweet's first slate read as too cute (rose-sniffing, blown kisses) and was rebuilt around magical-spectacle thorn-bloom power moves. Princess Arrogant's first slate read as too haughty-comedy (snap-shut-snap-open attitude beat as P2, dismissive eye-rolls as personality peaks) and was rebuilt around imperial-power moves with hero-tier weight + multi-component VFX. **Authoring rule**: every princess-class power slate must lead with **powerful, weighted, hero-tier moves** where the personality reads through *imperial bearing and certainty of her own power*, not through *haughty dismissal or comedic attitude beats*. Apply the General P6 v3 weight pattern (slow heavy committed wind-up → held charged apex → explosive strike → held imperial follow-through → slow regal return) on every power concept. The "smug knowing smirk" framing (she's *certain* the cut lands) is the right tonal target — never "cool dismissive smirk" (she doesn't bother to engage). No-force theatrical concepts (snap-attitude, fan-flutter, chin-lift dismissals) belong in **expressive idle** slots, never in the power slate. Audit every princess concepts.md against this before locking — if ANY power concept reads as "attitude-only" or "no real strike," replace it with a hero-tier weapon-action concept. Same audit applies to other heroes whose canonical pose carries inherent attitude (haughty nobility, sneering villain, smug merchant): the attitude is decoration on the hero-tier strike, never the substance of the strike itself.
- **Power move = weapon action, not ability (client direction, 2026-04-22)**: The power movement must be a clean beat of the character using their weapon — bow loose, hammer smash, pistol fire, fans snap, scepter thump, spoon club, scroll scribble. Do NOT design power moves around the hero's skill/ability kit (summoned walls, protective scrolls, thorn magic, AoE buffs) — those belong in gameplay, not the hero screen. Pull the weapon and its natural action from the hero doc's `Weapon` section and the CSV `Default Attack Style` / `Silviu notes`. Personality comes through *how* the weapon is used (stance, timing, expression, VFX), not through non-weapon substitutes.
- **Moneybags exception (client direction, 2026-04-22)**: Moneybags is the only hero whose power move combines two beats — he tosses his coin bag up and catches it, then raises and fires his pistol. All other heroes use a single weapon action for the power move.
- **Every hero has one mandatory breathing-only idle (client direction, 2026-04-22)**: The idle slot must always include one baseline option that is strictly breathing — chest rise/fall, a micro weight shift, minimal secondary motion (cape/hair drift, blink). No character business, no fidgets, no prop interaction. This is the safe fallback option the team can always pick. Expressive idles (Spy glance, Fat King belly pat, Moneybags coin heft, etc.) are additional variants proposed *alongside* the breathing baseline, never instead of it. See `PromptTemplates/animation_concepts.md` §"Idle Movement Concepts" for the I1/I2+ structure.
- **No blue rim light**: Earlier examples had a soft blue rim/edge light on character silhouettes (to composite onto the blue in-game background). This direction has been **reversed** — new animations should NOT include blue rim lighting. Reference animations in `Examples/` predate this decision and still show the old rim; ignore that detail when drawing inspiration from them.
- **Do not describe source-frame lighting in prompts**: The source PNG's lighting (warm key from upper-left, painted highlights) is already baked in and transfers automatically via I2V. Re-describing it in the prompt ("warm key light from upper-left", "warm lighting", etc.) introduces a second conflicting light rig and degrades the output. Only describe lighting when a new light source appears as part of the action — for example, a blue magic flame that ignites and illuminates the character, or a blade that glows on a slash.
- **Classify armor vs held prop during Stage 1 — they need different prompt language** (learned 2026-04-29 on Dragonwitch). Original Dragonwitch hero doc described the silver horned dragon skull as a **held mask totem in the left hand at hip level**. Re-reviewing the source art during Stage 1 with the user revealed it is actually **worn shoulder armor on her anatomical-left shoulder**, with that hand resting on the hip free of any prop. The two readings produce very different prompts: a held-prop version requires "always-hold" Constraints clauses (Fat King lesson) and risks Seedance dropping/re-gripping the prop mid-shot, while a shoulder-armor version locks the prop as a static body-attached element with subtle eye-glow VFX permitted on power beats. **Authoring rule**: during Stage 1 character review, audit each shoulder-region or hip-level element on the source art and explicitly classify it as one of `worn armor (body-attached)`, `held prop (hand-attached)`, or `magical effect (mid-air)`. Document the classification in the hero doc's `Costume & Equipment` section. Misclassification produces prompts that cannot be repaired by Stage-3 wording fixes — the only fix is to redo the description and regenerate.
- **Chroma background defaults to GREEN — only use blue when the character has green clothing/VFX** (learned 2026-05-06 on Moneybags). Defaulting to blue chroma without checking caused Moneybags (no green clothing, no green VFX, gold-only effects) to be queued on blue, which then bled blue tint into VFX-heavy beats (P2 slow-motion freeze frame turned blue). **Authoring rule**: at Stage 1, ALWAYS ask the user explicitly which chroma background applies — `Will this hero use green chroma (default) or blue chroma (only for heroes with green clothing/VFX)?` Default is **green (0x00FF00)**; use **blue (0x0000FF) only if** the character has green clothing, green VFX, or a signature in the green/yellow-green/teal hue range (Dragonwitch, Princess Sweet, Spy poison, etc. — see "VFX color collision" lesson below). The chroma color must match the FFLF background — `_FFLF_0.png` is the canonical green-chroma FFLF, `_FFLF_1.png` is the optional blue-chroma variant. Document the decision in the hero's `concepts.md` and apply consistently across all prompt files. Do NOT assume blue is the safe default — green is the default; blue is the exception for green-VFX heroes only.
- **Magic prop reformation breaks the I2V illusion — props returning to canonical must be physically caught, not sparkle-restored** (learned 2026-05-06 on Moneybags P2). When a held prop bursts / opens / disperses mid-loop and needs to return to its canonical state for FFLF integrity, the loop-close mechanism must be a **physical reunion** — the prop falls back down and is caught, the character reaches up and re-collects it, scattered pieces visibly reassemble in-hand, etc. **Magical sparkle-reformation reads as a Seedance asset-pop and breaks immersion** — the bag is gone, then warm gold sparkle fades and the bag is suddenly back; the eye reads it as a glitch, not a magic restoration. Moneybags P2 first-pass closed the loop with `the brown leather coin pouch reforms fully intact in his left hand at hip height with the drawstring tied closed` after the burst; user pushback was that the bag must NOT reform from nothing. Fix: the empty pouch falls back down through the air toward him from above and his left hand reaches up and catches it cleanly into his palm with a wordless cackle, his thumb pulls the loose drawstring tight in the same motion. Same rule applied retroactively to P3 v2 (coins escape off-frame and cannot return per user direction; magic-sparkle pouch restoration was kept only because the coins themselves are gone, and the **character's smug-mask reset and dignity-restoration beat** was added to carry the recovery so the magic is not the only thing the eye lands on). **Authoring rule**: at Stage 2 / Stage 3, when a concept involves a prop bursting / opening / dispersing, design the loop-close as a physical catch / re-grip / re-collect beat. Reserve magical-sparkle restoration only when the prop's contents genuinely cannot return, and pair it with a character beat that carries the recovery (mask reset, composed-grin, satisfied head-nod). Add to the negative prompt: `pouch reforming from nothing, pouch materializing from sparkle` (or the prop-equivalent). Princess Sweet's thorn-vine recede (vines retract back into the rose stem they grew from) is the canonical **good** version of this — VFX goes back the way it came, no asset-pop. Same principle: VFX/props should always have a continuous path from peak back to canonical, not a teleport.
- **VFX color collision with green chroma — generate a parallel blue-chroma variant for keying** (learned 2026-04-30 on Dragonwitch). Dragonwitch's signature dragon-flame VFX is canonically green; we re-tinted it to **greenish-purple** in prompts (per the standard "no green VFX on green chroma" rule from Spy P4). The greenish-purple flame still survived keying badly enough that the keyer ate the green channel of the flame on most clips. **Two-fix pattern that worked**: (a) keep the **green-chroma generation set** as the primary "show the AD" deliverables (Seedance generates cleaner facial detail and edge work on the trained-default green chroma); (b) generate a **parallel blue-chroma variant set** by swapping the chroma constraint from `pure green chroma key 0x00FF00` to `pure blue chroma key 0x0000FF` and using a blue-background FFLF source PNG (`<Hero>_FFLF_1.png`) — this gives the keyer clean separation from the greenish-purple flame. **Authoring rule**: any hero whose canonical signature VFX sits in the green/yellow-green/teal hue range (Dragonwitch dragon-flame, Princess Sweet thorn vines, future heroes with ivy/poison/forest-magic VFX) should ship with **both a green-chroma and a blue-chroma prompt variant** generated from the start. Maintain the variant as a `<ConceptId>_1_<Name>_BlueScreen.md` prompt file in `Prompts/Seedance/` with a corresponding `<Hero>_FFLF_1.png` blue-chroma source PNG. The variant adds `green background, green chroma` to the negative prompt as belt-and-suspenders. Heroes with red/orange/blue/purple VFX (most of the roster) do not need the variant.
- **Garment-noun over-read family — extend to "tattered skirt"** (learned 2026-04-29 on Dragonwitch, family of `mantle → cape` and `veil/streamer → trailing banner`). The phrase `tattered skirt` collapses to a longer flowing silhouette in Seedance output even when the source art shows a short ragged skirt. **Fix**: describe concretely — `short green leaf-shaped skirt with ragged jagged edges falling to mid-thigh`. The full-descriptive phrase locks the silhouette length and the edge-shape language ("ragged jagged edges") covers the tattered intent without the over-read trigger. **Authoring rule**: during Stage 1 / Stage 3, audit any garment word that has a longer-silhouette default in fashion vocabulary — `tattered skirt`, `flowing skirt`, `gown`, `train`, `cape`, `cloak`, `mantle`, `robe`, `veil`, `streamer`, `sash`. Replace with a concrete phrase that locks length (`falling to mid-thigh`, `ankle-length`, `at the knee`) and shape (`leaf-shaped`, `tiered`, `ragged jagged edges`).
- **Use a mouth-closed FFLF variant when the canonical pose has an extreme expression** (learned 2026-04-23 on Fat Princess). When a hero's source PNG shows an extreme/peak expression — wide-open-mouth surprise, mid-laugh roar, fully-closed-eye bliss — that expression is fine for a static hero card but reads as awkward when held for the full 4-15s loop duration as both first and last frame. The fix is to maintain a **mouth-closed (or otherwise neutral-expression) variant** of the source PNG and use it as the I2V input instead. The extreme expression then lives mid-loop as a peak beat, with the loop opening and closing on the stable resting state. **Authoring rule for new heroes**: during Stage 1, evaluate whether the canonical source PNG's expression is "neutral resting" or "extreme peak." If extreme, request or create a neutral variant (closed-mouth smirk, calm gaze, etc.) and document it in the hero doc's header as the FFLF input. Existing variants live under `Source/Heroes Stylized/mouth_closed/`. Worth checking other heroes for the same need (any open-mouth grins, mid-action faces, etc.). Hero docs should call this out explicitly at the top: "FFLF image convention: use the [neutral variant] as the I2V input."
- **Off-hand prop integrity is an FFLF constraint, not a soft preference** (learned 2026-04-23 on Fat Princess). When the canonical pose includes a held prop with a distinctive shape (Fat Princess's intact yellow cupcake, a sealed scroll, a full bottle, a coin in a specific orientation), the prop must appear identical at t=0 and t=loop-end because both frames are the same source image. This forecloses entire concept families — no bites, no crumbs, no frosting reveals, no pour-outs, no broken seals. **Authoring rule during Stage 2**: before proposing any concept that interacts with an off-hand prop, check whether the interaction visibly mutates the prop's appearance. If yes, the concept is invalid as written. Workarounds: (a) hold/admire/wave/gesture without consumption; (b) cycle the prop out and back via a comedic refill gag (Fat Princess P2 — bat the cupcake away, pull a fresh one from a hidden gown pocket); (c) reframe the beat as a held-pose theatrical reveal rather than an action that mutates the prop; (d) **throw + return** mid-loop — boomerang, toss-and-catch, hurl-and-recover (validated 2026-05-06 on Princess Arrogant fan-boomerang concepts). Throw + return is a fully valid concept family on heroes whose canonical prop is durable and where it's plausible the prop returns to canonical hand position before loop end. The integrity rule is "prop is identical at t=0 and t=loop_end" — *what happens between those frames* (including being briefly off-hand or off-frame) is unconstrained, as long as the prop returns to canonical position with canonical shape. Constraint clauses must spell out the return: `<prop> returns to <hand/position> by end` at start of Constraints (replaces the standard `always-hold` clause for the throwing hand only — the non-throwing hand still uses `always-hold`). The Moneybags coin toss-and-catch is the existing precedent; Princess Arrogant generalized it to single-fan and twin-fan boomerangs. Document the integrity constraint at the top of the hero doc when relevant.
- **Audit the roster for kinetic-shape collisions during Stage 2** (learned 2026-04-23 on Fat Princess). Multiple heroes converge on similar power-move silhouettes by default — the "raise weapon overhead, smash down" pattern alone is shared by Blacksmith P1 (hammer ground slam) and New King P3 (sword overhead rally + chop). When proposing power concepts for a new hero, sweep the locked Approved Concepts in existing `Output/<Hero>/concepts.md` files and **diversify the axis-of-force**: vertical down (smash), horizontal swing (sweep / spoon-bat), forward thrust (jab / fire / point), rotational (pirouette / fan-snap / coin-twirl), no-force theatrical (showcase / lick / salute). Five distinct axes is plenty across the roster. The audit is also prompt-craft insurance: when two heroes share a kinetic shape, the second one is harder to make distinctive and risks reading as a clone. Keep one fallback copy of the shared-pattern concept on the candidate list (Fat Princess P1 still exists as Overhead Spoon Smash) but recommend a different primary.

### Webapp Job Submission Conventions

- **Output filename pattern is `<conceptId>_<version>.mp4`** — the webapp auto-increments `<version>` on each new generation queued under the same `conceptId`. First run = `_0`, second = `_1`, etc.
- **Variant prompts (different chroma background, different VFX color, different FFLF, etc.) are NOT separate concepts** — they live as additional prompt files in `Output/<Hero>/Prompts/Seedance/` and are selected in the webapp UI as the prompt-file source for a normal `conceptId=<parent>` queue.
- **Authoring rule** — when queueing a variant programmatically (curl to `/api/jobs/seedance`), set `conceptId` to the **parent concept** (`P1`, `P2`, `I1`, etc.) and `promptFile` to the variant file (e.g. `P1_1_Mocking_Flame_Hurl_BlueScreen.md`). The webapp will then write the result as `P1_<next_version>.mp4`. **Do not** set `conceptId="P1_1"` — that produces a malformed `P1_1_0.mp4` filename which the UI won't parse.
- **Variant prompt-file naming** — `<ParentConceptId>_<index>_<Name>_<Tag>.md` is the convention. E.g. `P1_1_Mocking_Flame_Hurl_BlueScreen.md` means "P1 variant index 1, blue-chroma version." The `_<index>` in the filename is purely a way for humans to keep variants apart — the webapp parses the conceptId prefix (`P1`) for grouping and shows the file as a selectable prompt under P1.

### Seedance 2.0 I2V Pitfalls (learned 2026-04-16 on Spy)

- **Lead the Subject line with "static camera shot of..."** whenever a concept has high motion or off-screen beats (kicked bombs, fired bolts, sharp head snaps, etc.). Without this prefix, Seedance interprets the dynamic action as an invitation to move/zoom the camera. User-validated on P1 Trollface and P5 Bomb Mishap.
- **Never write "whole upper body leans in" / "leans forward toward the viewer"** — Seedance reads this as a camera zoom-in rather than a character lean. Express forward motion as a character-internal beat ("he leans his shoulders and head forward a few inches", "he tilts his head in conspiratorially") and pair it with the static-camera prefix.
- **No green-colored VFX on the green chroma background.** Glowing green auras, sickly-green potions, green magic sparks — all collapse into the chroma key and get eaten by keying. Switch to purple, orange, red, blue, etc. Confirmed on Spy P4 (poison vial was originally green, switched to purple with skull sign on the bottle for unambiguous "poison" read without fighting the keyer).
- **Avoid rapid chained head snaps on characters with back accessories** (capes, quivers, hoods, banners). Seedance will morph or duplicate those accessories during the snap chain — fletched arrows mutate, phantom capes appear on characters that don't have one. Confirmed by I5 Target Double-Take (Spy, 0/2 keeps due to arrow morphing and phantom cape swing). One sharp head snap is fine; a double-take chain is the risk zone. If the concept demands chained snaps, prefer characters with tight silhouettes.
- **Prefer sustained clear actions over minimal-beat fidgets.** "Pat pat pat on the satchel" produced flat, under-motivated frames; reworded as "rummages in the satchel" the same concept produced richer motion and better expression coverage. Principle: I2V fills its generation budget better when given a clear sustained action than a list of small tics. See also `Docs/Heroes/Spy.md` "I2V Motion Granularity" for the character-level version.
- **Prompt redundancy helps ban-intent stick.** Bracket the Sound line with `no voice no dialogue no music` at both ends of the list, and lead the Constraints line with `no dialogue`. Doubling the ban across fields reliably suppresses lipsync and incidental voice noise that Seedance otherwise sneaks in.
- **Hand convention = anatomical (character's own left/right), not viewer-side** (learned 2026-04-16 on Fat King). When we write "left hand" / "right hand" in a prompt, Seedance interprets it as the **character's anatomical** hand — not the hand on the left/right side of the image from the viewer's perspective. Because most hero source art is in a 3/4 view, the viewer-left side of the image usually corresponds to the character's anatomical right. **Authoring rule**: for each hero, determine which hand holds the weapon/signature item and which rests on the hip in anatomical terms, document it in the hero doc's `Pose Description`, and use the anatomical terms consistently across all prompts. Example: for Fat King, the scepter is held in his **left** hand (viewer's right side of image) and his **right** hand rests on the hip (viewer's left side of image). Conflating viewer-side with anatomical in a prompt causes Seedance to place the wrong hand on the wrong object (e.g. the scepter floats, or the character switches hands mid-shot).
- **Avoid "rolls"/"rolling" when describing character body motion with weight-shifts or hip/belly phrasing** (learned 2026-04-16 on Fat King I4). Phrases like "his body rolls gently to his left" or "heavy rolling rhythm" combined with `belly`, `hip`, or `weight shift` can trip Seedance's text safety filter and silently reject the job — the request returns status COMPLETED with a ~0.05s `inference_time`, empty `logs` array, and the result endpoint returns 400 "There was an error parsing the body." Content-wise the prompt is benign; the filter is over-tuned. **Fix**: use `sways`, `rocks`, `leans`, or `shifts` instead of `rolls` for weight-transfer body motion. "Rocking rhythm" > "rolling rhythm"; "sways his torso" > "body rolls." Characters doing physical rolling motions (acrobats, falls, tumbles) would genuinely need "roll" but hero-idle weight-shifts do not.
- **The `roll` filter trigger extends beyond body motion — high `roll`-family density anywhere in the prompt can also reject** (learned 2026-04-27 on Architect P2). The Architect Blueprint Tap concept used `roll`-family verbs purely for parchment scrolls (`unrolls`, `rolled scrolls` ×3, `rolls back up`, `re-roll crinkle`) — no body phrasing — and still hit the same silent rejection (COMPLETED, ~0.05s inference_time, 400 "error parsing the body" on result). The filter appears to keyword-match `roll`/`rolled`/`rolls`/`unrolls`/`re-roll` density rather than inspect semantic context. **Authoring rule**: for parchment, scroll, paper, or any `roll`-shaped object actions, use **`coil`/`coiled`/`unfurl`/`unfurled`/`re-coil`** as drop-in synonyms. "Three coiled scrolls" > "three rolled scrolls"; "the scroll unfurls and coils back" > "the scroll unrolls and rolls back." `Rolled-up sleeves` is a fixed compound phrase that has not yet tripped the filter and can stay (it appears in every prompt for the Architect's tunic) — but if a future rejection appears with even cleaner language, suspect this phrase next.
- **Negated violence words can still trip the filter — keyword filters often don't parse negations** (learned 2026-04-27 on Architect P2; **confirmed extends to idles 2026-05-06 on Diana I2**). The original Architect P2 Constraints line included `not a stab not a strike the parchment is never torn never cut never pierced` as a defensive list. Even though every word was negated, the raw tokens `stab`/`strike`/`torn`/`cut`/`pierced` registered with the filter and contributed to (or caused) the rejection. The Diana I2 (Sword Shoulder-Tap idle) recurrence confirms this generalizes from non-combat power moves to **idle prompts**: I2's Constraints had `the two taps are light subtle taps with the flat of the blade not a heavy slap not a strike not a cut against the shoulder` — a benign blade-touch idle, but the negated `slap`/`strike`/`cut` cluster paired with weapon-noun + body-part triggered the same silent rejection. **Authoring rule**: never list violence-adjacent words in negation form anywhere in the prompt — power moves, idles, or any prompt type. Reword as a positive statement: "the two contacts are light gentle touches of the flat broad face of the blade against the fur collar" instead of "not a heavy slap not a strike not a cut against the shoulder." Same principle applies to weapons, body parts, or any other word family the model might safety-filter — express what you DO want, not a list of what you don't.
- **The `impact` word family is a standalone text-filter trigger, even in legit combat prompts** (learned 2026-05-06 on Diana P1 Flourish Slash & Fist Pump). Architect P2 first surfaced `impacts` as one component of a dense weapon-action cluster, but Diana P1 isolates `impact`-family density as a **standalone trigger** in an otherwise textbook combat-power prompt. P1 first-pass had `slash impact midpoint`, `slash impact frame` (in secondary motion), `the moment of the slash impact` (Sound), and `fist-impact-air whoosh` (Sound). Same silent rejection signature (COMPLETED, ~0.05s `inference_time`, 400 "error parsing the body"). P3 (Diagonal Upcut Fist Punch — same combat-power kinetic class with battle-cry shout, slash, and fist-punch language but **no `impact` words**) ran cleanly. **Authoring rule**: avoid `impact` / `impacts` / `impact frame` / `at the moment of impact` / `fist-impact` / `slash impact` anywhere in any prompt type. Drop-in replacements that worked on Diana P1 v2: `peak` (`slash midpoint`, `slash peak frame`, `peak of the slash`); `air whoosh` (instead of `fist-impact-air whoosh`); `appears at` (instead of `pops at` / `popping at`). The filter appears to keyword-match the `impact` token regardless of semantic context — same shape as the `roll`-family trigger. Apply on every combat-power prompt by default.
- **Body-organ anatomy words trip the text filter, even as a vivid metaphor** (learned 2026-05-08 on Princess Arrogant I1 v3). The skirt-locking rule was anchored with the memorable line `the skirt does not act as an additional pair of lungs` to prevent the breath-rhythm pulsing the AD had flagged. Same silent rejection signature (HTTP 400 "error parsing the body") despite the line being a benign metaphorical negation about fabric motion. The trigger was the **organ-noun `lungs`** — a body-part anatomy token, same family as the negated-violence and `impact` keyword filters. **Authoring rule**: avoid body-organ / internal-anatomy nouns anywhere in any prompt — `lungs`, `heart`, `liver`, `stomach`, `intestines`, `bones`, `skull`, `flesh`, `veins`, `nerves`. Skin / hair / face / hands / chest / shoulders / torso / hips / legs / feet are all fine — the filter is on **internal organ + skeletal** tokens, not body-surface nouns. Memorable rule-anchor lines (e.g. "the skirt does not act as X") are otherwise excellent prompt craft and should keep being used — just choose vocabulary outside the filter's keyword list. Drop-in replacement that worked: `the skirt does not pulse or expand and contract in time with the breath rhythm`. Same shape as the `impact` and `roll` family triggers — keyword match regardless of semantic context.
- **Targeted-fix rule — never cascade lockdowns when AD flags a single specific behavior** (learned 2026-05-08 on Princess Arrogant I1 v2/v3). When the AD note was "skirt expands and contracts as she breathes, would adjust that so she breathes through her chest," v2 layered `absolutely still` / `imperceptibly` / `locked` language across hair, sleeves, earrings, fans, posture, AND skirt — producing a near-frozen output the user described as "barely any movement." The user's correction was the rule: **"a natural relaxed idle, but limit the skirt movement"** — i.e. fix the named element, leave everything else as a normal idle. **Authoring rule**: when iterating on AD feedback for a specific behavior, the prompt edit should be **surgical** — touch only the language that controls the named element (here: the skirt secondary-motion clauses and one targeted Constraint line about skirt-pulsing). Do NOT add blanket lockdown clauses on adjacent elements ("everything else stays absolutely still") as a defensive over-correction. The instinct to over-tighten comes from wanting the regen to not surface a different complaint — but it produces dead frames that themselves become the next complaint. Same principle applies to unit-progression edits when the AD flags one detail (a buckle, a pauldron rim, a cape-edge color) — fix only that, don't tighten the whole prompt. Cross-reference with the existing "Detail Economy" rule (cut surface ornament, don't stack) — same family of "less is more, named only."
- **Re-noun weapons as instruments when the action is non-combat** (learned 2026-04-27 on Architect P2 — third rejection after the `roll` and negation cleanups). Even with `roll`-family and negated-violence words removed, P2 still hit the same silent filter rejection. The remaining trigger was the **dense semantic cluster of weapon/violence-adjacent words around the blade-on-parchment action**: `blade` (×6), `tap`/`tap-tap` (×4), `impacts`, `burst` (in `ripple-burst`), `sharp`, `contacts`. The action was semantically a **measurement tap** on a paper diagram, but the vocabulary read as combat. **Fix**: re-noun the "weapon" as a **measuring instrument** for the specific concept where it isn't being used as a weapon — `silver drafting calipers` instead of `silver caliper-protractor blade`. Replace `tap`/`taps` with `touch`/`touches`, `impacts` with `tick sounds`, `ripple-burst` with `ripple-glow`. Drop `sharp` and `contacts` from action vocabulary. **Authoring rule**: when a hero's weapon is functioning in a **non-combat capacity** in a specific concept (a sword being inspected at the hilt, a hammer resting on a shoulder, a bow being polished, a caliper-blade tapping a diagram), audit whether the weapon-noun + action verbs create a false combat semantic field. If yes, re-noun the object as its civilian-role analogue **for that prompt only** — keep the weapon-noun in concepts where the action genuinely IS a strike/slash/swing.
- **Prepend a short "always-hold" clause to Constraints when the canonical pose includes a held prop** (learned 2026-04-16 on Fat King). When a hero's canonical pose has them holding a weapon/scepter/tool and the prompt involves fidgets or expressive beats that don't involve the prop, Seedance often mid-shot releases, drops, or re-grips the prop onto the wrong hand even when the Action/Notes explicitly say otherwise. **Fix**: lead the `Constraints:` line with a short imperative like `one hand on golden staff always,` or `crossbow stays in left hand always,` — keep it under 7 words and put it at the start of the constraints list where it carries the most weight. This is redundant with the longer "scepter stays upright in left hand throughout never dropped" clause later in the same line, but the redundancy meaningfully improves adherence (same principle as doubling the `no voice no dialogue no music` ban). Name the prop with a simple, common noun (`staff`, `hammer`, `bow`) even if the hero doc uses a more specific term — Seedance responds better to short direct words here.
- **"Mantle" reads as "cape" — use "scarf" or "stole" for ermine/fur neck-pieces** (learned 2026-04-16 on Fat King). The word `mantle` collapses to a full draped cape/robe silhouette in Seedance output, even when the source art clearly shows a smaller shoulder-fur or ermine collar. Fat King's ermine neck-piece was originally described as "ermine mantle" and Seedance gave him a full cape every time. Switching to **`ermine scarf`** (or the historically-accurate `ermine stole`) fixed it — scarves/stoles have unambiguous small-neck-piece semantics. **Authoring rule**: never describe small fur shoulder-pieces or ermine collars as "mantle" or "mantelet" in prompts. Use `scarf`, `stole`, `fur collar`, or a fully descriptive phrase like "white spotted ermine fur around the neck and shoulders." Same principle applies to any other garment-noun that Seedance over-reads into a larger silhouette (e.g. `cloak` for a cowl, `robe` for a tunic).
- **"Twirl" / "spin" reads as a full body rotation, not an object-only rotation** (learned 2026-04-23 on Fat Princess P4). When the prompt said "the spoon handle rotates once in her palm" plus "one clean horizontal twirl," Seedance generated a full-body pirouette where the character turned 360° in place while the spoon swept a chocolate ribbon around her silhouette. The user approved the body-rotation read and the concept was rebaked around the pirouette. **Authoring rule**: if you want a hand-only object flourish (say, a knife spun in the fingers), avoid `twirl` / `spin` entirely — use `the spoon handle rolls across her fingers` or `she flips the knife once in her hand`. If you want a body rotation, be explicit: `turns exactly one complete 360-degree pirouette in place, ends facing forward.` Two protective clauses are needed either way: (a) `exactly one complete rotation` to prevent chained spins; (b) `camera does not orbit only the character turns in place` to prevent Seedance from reading the rotation as a camera orbit around a static subject.
- **"Veil" / "streamer" on conical hats reads as a long trailing banner — use "short white cloth tassel at the tip"** (learned 2026-04-23 on Fat Princess). Same family of over-read as `mantle → cape`. Fat Princess's tall pink conical hat has only a **short white cloth tassel** (a few inches of fabric) at the tip in the source art, but Seedance consistently extended it into a long flowing veil / trailing banner silhouette down her back when the prompt used the words `veil` or `streamer` — even with modifiers like `small` or `short`. The fix is to **drop both nouns** and describe the element concretely as `a short white cloth tassel at the tip of the hat`. Both the positive description AND an explicit negation in the Constraints line (`hat has only a short white cloth tassel at the tip not a long flowing veil or trailing banner`) improved adherence; relying on either alone was less reliable. **Authoring rule**: audit any conical / pointed / medieval hat description for `veil` or `streamer` — prefer `tassel`, `tuft`, `pennant`, or a plain descriptive phrase ("a small triangular cloth flag attached to the hat tip") depending on the actual shape. Secondary-motion phrasing should refer to "the hat tassel" — never "the veil drifts / streamer sways."
- **VFX intricacy upgrade — "elaborate ornate diagram with rich ornamental detail" pattern** (learned 2026-04-27 on Architect P1 + P5). When a first-pass VFX generation reads as too basic / boring (a single thin trail, a plain ring, a generic cross-hair, a "speedometer-style" arc), the fix that worked twice in a row was: (a) describe the VFX in the Action body as `an elaborate ornate diagram trail` with **rich ornamental detail**; (b) name 4-5 specific concrete elements layered into the figure (`multiple fine layered arcs at smaller radii nested inside the primary sweep, fine tick-mark notches at regular intervals along all the curves, small dot accents spaced along the trail, short radius lines from a central focal point, and a small triangle of fine line work near the center`); (c) **join the elements as connected prose with conjunctions** — never an em-dash list of small noun phrases (per the Section A5 trigger below); (d) close with a comparison anchor like `like a detailed ornate protractor diagram with rich ornamental detail drawn in midair`; (e) add an explicit Constraints negation of the simple-shape fallback (`not a single simple speedometer-style arc` / `not a single simple ring or basic cross-hair`). **Stay within safe-zone vocabulary** — `elaborate`, `ornate`, `detailed`, `rich ornamental detail`, `protractor diagram`, `target marker`, `blueprint` all worked; avoid `compass-and-dividers`, `construction circles/triangle`, `vesica-piscis`, `network`, `draftsman's working diagram`, `constructs itself` (each was suspected of triggering the safety filter on Architect P1 v2/v3 rejections).
- **Never include literal quoted personality phrases in the Action body — they cue the audio model to generate actual speech** (learned 2026-04-28 on Architect P10 v4). The Action body had `with smug "I have unveiled my masterworks" verdict` as a personality cue. Even though Sound bracketed `no voice no dialogue no music` at both ends and Constraints said `no dialogue`, the output audio contained actual speech at the verdict beat. The audio model reads Action as scene context and a literal quoted phrase tells it what speech happens at that beat — overriding the prompt-level vocal ban. **Authoring rule**: express personality through facial expression / pose / posture / head-shake / brow-arch in Action, never through quoted text. `with smug verdict and a small dismissive head-shake` not `with smug "I have unveiled my masterworks" verdict`. **Related: extend the standard Spy `no voice no dialogue no music` ban to `no voice no dialogue no speech no vocalizations no humming no breaths no music` for any prompt with a verdict / proclamation / commanding-gesture beat**, and add an explicit positive Constraint `no dialogue no speech no vocal sounds in audio only material and mechanical sounds like [list] are present in the audio track` (Section A6 family 1 pattern from Princess Sweet, now extended to verdict-beat scenes).
- **Avoid nose-breath items in Sound — they render as snivelling/sniffing audio** (learned 2026-04-28 across Architect P6/P9/P10 first-pass outputs). Items like `a soft refined nose-inhale on the chin lift`, `a small refined nose-exhale on the settle`, `a satisfied nose-snort on the verdict head-shake` produce sharp inhalation/exhalation/sniff sounds in the generated audio that read as snivelling or sniffing in playback. The text safety filter accepts these items and the output-audio safety check (Section A6 in `Docs/PromptGuides/Seedance/seedance2_content_flags.md`) also passes them, but the **audio quality** is poor. **Authoring rule**: drop all `nose-inhale`, `nose-exhale`, `nose-snort`, `nose-breath`, `nose-chuckle` items from the Sound field. Audio coverage of moments like "chin lift," "verdict head-shake," "settle," etc. should come from **mechanical / material / fabric sounds incidental to the body motion** instead — `a soft fabric shift of the green tunic on the chin lift`, `a soft cloth rustle of the red turban side-tail on the head-shake`, `a faint metallic ambient shimmer from the blade on the settle`. Cross-reference with Princess Sweet P1/P6 incidents (Section A6 family 1) which had a similar finding scoped to body-contact + intimate-vocabulary scenes — the Architect 2026-04-28 finding extends the rule: nose-breath items are quality-poor across **all** scene types, not just intimate ones, so they should not appear in any prompt's Sound field by default. Non-verbal mouth sounds that are NOT nose-breath (e.g. a clean wooden-leather catch snap, a metallic blade tick, a parchment crinkle) remain fine.
- **Seedance defaults to continuous circular/arc motion when given "weapon flourish" prompts** (learned 2026-04-27 on Architect Stage 5 review). The model has a strong prior for smooth continuous arc patterns and will collapse multi-stage discrete actions into one continuous circular motion if the prompt isn't explicit. **Visual symptom**: a hero's first 3-5 power-move generations all read as variants of the same circular flourish even when concepts were intentionally distinct. **Two-level fix**:
  - **At the concept stage**: diversify kinetic shape within the hero's own set (see `PromptTemplates/animation_concepts.md` per-hero kinetic-shape diversification rule). Don't let all concepts converge on horizontal arc + rotational flourish + forward press patterns.
  - **At the prompt stage when a concept genuinely IS multi-beat** (throw → review → catch; raise → plant → hold; pull-back → cut → settle): enumerate each beat as a **discrete timecoded segment** in the Action body — `(0.0-1.0s) he raises the blade overhead, (1.0-1.6s) he plants it sharply downward, (1.6-3.0s) he holds the planted pose with a smug verdict head-shake, (3.0-4.5s) he pulls the blade upward and rotates it back across his body, (4.5-6.0s) settles to canonical`. Discrete beats need discrete punctuation. Avoid run-on Action bodies for multi-beat concepts — the model will smooth them into one circular motion if you let it.
- **Output-audio safety rejection is a separate failure mode from text-filter rejection** (learned 2026-04-27 on Princess Sweet P1, then P6 v1→v5 over 2026-04-27/28). When the rejection signature is `httpCode: 422` + `inference_time` ≈ 150–190s + `"Output audio has sensitive content"` + `reason: partner_validation_failed`, the **video generated successfully** but the auto-generated output audio failed a partner-validation safety check. Two trigger families: **(1) body-contact + intimate-vocabulary** (vines wrapping body + warm/soft Sound modifiers + nose-exhale → audio model generates breathy/sigh sounds → reject); **(2) mood-music inference** (chime/musical-instrument words + serene/calm/warm Action mood modifiers + spectacular VFX → audio model generates orchestral/musical content → music classifier reject). Fix patterns differ: midair-only VFX + mechanical Sound for (1); drop chime/musical words + drop ALL warm/serene Action modifiers + explicit `no music no orchestral no swells no melodic tones` ban + reduce duration for (2). Always grep for orphaned `warm`/`soft` after multi-edit sessions — duplicate Expression-arc blocks linger and reintroduce the trigger. Full failure-mode docs and worked examples live in [seedance2_content_flags.md A6 + C.2](Docs/PromptGuides/Seedance/seedance2_content_flags.md). For a princess-class power move with body-contact VFX or magical-reveal spectacle, **preemptively apply both fix patterns from prompt v1** — don't wait for the rejection.
- **I1 breathing baseline — frame as continuous ambient rhythm, not discrete cycles** (learned 2026-04-30 on Princess Sweet I1, propagated to Dragonwitch I1). The earlier Architect-era I1 framing of "calm regular breaths with multiple small breath cycles" still produced visibly heavy chest motion that read as over-pronounced breathing on the final loop. The fix is to (a) re-frame in Action as **continuous ambient breathing with no discrete visible breath events**: `her breathing is slow and calm and ambient with the chest moving gently and almost imperceptibly throughout the loop in a continuous smooth rhythm without any visible discrete breath events`; (b) drop "with each chest rise" / "in time with each breath" secondary-motion language and replace with `drifts almost imperceptibly with the ambient breathing` so secondary motion does not amplify the breath read; (c) lead the Constraints line with the **full ban list**: `no deep breaths no sharp breaths no quick breaths no short breaths no panting no audible inhales no audible exhales breathing is slow and calm and ambient and barely perceptible like a person standing relaxed at rest`; (d) close Constraints with `breathing is a continuous smooth ambient rhythm with no discrete visible breath events` (doubling the ban across Action and Constraints per the standard redundancy lesson); (e) Sound field uses `soft natural calm ambient nose-breathing at a slow steady relaxed rhythm` — drop "with multiple small breath cycles" and "in time with each breath" wording. **Authoring rule**: every hero's I1 prompt should use this updated pattern by default; the Architect I1 wording ("calm regular breaths with multiple small breath cycles") is **deprecated**. Worth back-porting on any I1 that still shows heavy breathing on review.
- **Surfaces that conventionally hold content default to BLANK unless the prompt specifies what content is on them** (learned 2026-05-05 on Herald P4/P5/P6 first-pass review). When a hero holds a **scroll, book, page, parchment, letter, map, banner, sign, tablet, ledger, or any prop with a visible surface that conventionally bears writing or imagery**, Seedance defaults to **rendering the surface blank** — no text, no symbols, no diagrams, no emblems — unless the prompt explicitly describes the visible content. This is especially destructive for: (a) bureaucratic / scribe / herald characters whose comedy depends on the document being a real document; (b) X-mark / cross-out / strike-through power moves where the X must land ON existing content to read as crossing-out; (c) reveal / unfurl / open-the-book power moves where the unfurled section is the spectacle. Symptom: scroll unfurls to show a smooth tan rectangle, the X-mark slashes across blank parchment, the book opens to empty pages, the banner unrolls into a featureless cloth. **Authoring rule**: at Stage 1 / Stage 3, audit every hero whose canonical equipment includes a content-bearing surface. **In the Subject line**, describe the surface as already covered with appropriate visible content — `tan parchment scroll covered in visible flowing handwritten black-ink calligraphy lines of royal-decree script`, `open leather-bound book with visible handwritten ink notation and small ink diagrams on the visible page`, `wide blue banner with a visible embroidered gold royal sigil at the center`, etc. **In the Action body**, when the prop unfurls / opens / reveals more surface area, explicitly describe the additional content revealed: `revealing additional lines of visible handwritten black-ink calligraphy on the longer exposed section`. **In the Constraints**, add an integrity clause: `the parchment has visible flowing handwritten calligraphy throughout the entire loop the parchment is never blank or empty`. **For mid-loop marks added by the action** (P1 fresh scribbles, P5 X-mark cross-out, etc.), distinguish the **fresh action-added marks** from the **canonical existing content**: the fresh marks fade before loop end so the FFLF first/last frame matches the canonical scroll content exactly. The integrity clause closes with: `and the original existing content on the parchment stays the same as the canonical pose at start and end`. Heroes immediately affected by this rule: Herald (scroll), Architect (blueprints / ledger), New King (royal proclamation / map), and any future hero whose weapon is a document, book, banner, or inscribed surface.
- **Distinctive weapon silhouettes morph during fast motion — broad-face presentation is the structural fix** (learned 2026-04-27 on Architect P4). When a hero's weapon has a **non-standard silhouette** (Architect's caliper-protractor with curved arc + calipered prongs + cutout handle is the canonical case; the same risk applies to Fat Princess's spoon-club, Princess Arrogant's fans, Dragonwitch's flame, Princess Sweet's vines, Spy's crossbow shape, Architect's articulated brass drafting arm, etc.), I2V models substitute a generic blade/sword/dagger/spear shape during fast motion when the silhouette is foreshortened toward camera. Three-part fix, applied in order:
  - **(A) Subject silhouette geometry** — describe the weapon's shape concretely in the Subject line (`a wide curved protractor arc as the bottom edge engraved with fine tick marks, two angular calipered prongs extending forward, and a circular cutout handle in the body of the blade`). **This is necessary but not sufficient on its own** — verified on Architect P4 v2. Treat it as a baseline anchor that helps but doesn't fix the morph.
  - **(B) Reframe the action to keep the broad face presented to camera** — the **structural fix**. Replace tip-first thrusts/jabs/stabs with broad-face presses (`presses the blade forward with the curved protractor arc face turned toward the viewer and the full broad face presented flat to the camera throughout the press`). **Verb choice matters**: `press` / `presents` / `displays` imply broad surface contact; `thrust` / `jab` / `stab` imply tip-first. Bonus: broad-face presentation often plays better with the personality beat than a tip-first stab — for refined / scholarly / ceremonial characters especially.
  - **(C) Explicit silhouette-preservation Constraint** — `the full silhouette including [list the distinctive elements] remains clearly visible to the camera at every moment of the loop and the silhouette never simplifies into a generic sword or spear shape`. Negation of the simplification fallback is explicit.
  
  **Authoring rule**: during Stage 3 prompt creation, audit every hero with a non-standard weapon and preemptively apply (A) + (B) + (C) to power-move concepts that involve fast or foreshortening motion. Don't wait for the first-pass morph to discover this.
- **Projectiles render in reverse orientation by default — explicit positive direction-of-flight description plus negation are both required** (learned 2026-05-04 on General P7 Volley Command). When a prompt has a projectile in flight (arrow, bolt, bullet, thrown spear, javelin, hurled axe), Seedance frequently renders it **flying tail-first** — the feathered/finned end leading and the tip trailing — even when the part anatomy is described correctly. Naming the parts is not enough; you have to anchor the **direction of flight relative to the parts**. **Authoring rule**: in Action and Constraints, write `every arrow flies point-first with the sharp metal tip at the front of the arrow leading the direction of flight and the feathers at the back of the arrow trailing behind the wooden shaft`. Pair with explicit negation `never flying backwards never tail-first never feather-first never reversed` in Constraints, and add `arrows flying backwards, arrows flying tail-first, feathers leading the arrow, tip pointing backwards` to Negative prompt. Same pattern generalizes to bolts (tip + nock), bullets (tip + base), thrown axes (head leads). **Avoid technical archery jargon** — `fletchings` underread by the model in early generations; switching to `feathers at the back of the arrow` worked. Plain everyday language for object parts beats domain-specific vocabulary across the board.
- **Off-screen origin needs depth-cue language, not just "behind him"** (learned 2026-05-04 on General P7 Volley Command). When something enters from off-screen, `from off-screen behind and above him` is ambiguous — Seedance often interprets `behind` as `to the side` and arrows curve in from screen-right or screen-left rather than emerging from depth. **Fix**: anchor origin in **vanishing-point depth-cue language** — `from the vanishing point in the deep background directly behind his back, small in the distance, clustered together at the vanishing point` — and frame the path as `fly forward toward the camera in straight parallel lines from back to front, growing larger as they come closer in depth`. Add explicit Constraint negations: `not entering from the left side of the frame not entering from the right side of the frame not curving sideways not arcing in from the side`. Also add `arrows from the right side, arrows from the left side, arrows entering from screen edge, arrows curving sideways` to the Negative prompt. Same pattern applies to anything entering from depth (incoming spell, summoned reinforcements, charging cavalry, falling debris): name the **vanishing point** as the origin and describe the path as **growing larger from back to front** in parallel lines, not as `arcing in` (which reads as a sideways arc).
- **Synchronized-timing language uses "AND" framing, not "THEN"** (learned 2026-05-04 on General P7 Volley Command). When a body action and an off-screen VFX should land at the same instant — chop fires the volley, swing releases the magic, slam triggers the shockwave — Seedance defaults to a sequential read where the VFX arrives noticeably after the body action completes if the prompt uses `then` framing or describes them as separate beats. **Fix**: write the timing as `at the apex of the chop / synchronized with the chop`, and structure the sentence so the body action and the VFX release happen in the same grammatical clause: `the bowstring snap fires at the apex of the chop and the leading edge of the volley is already streaming over his head as the chop reaches chest height so the order being given and the volley releasing happen in the same moment not one after the other`. The phrase `in the same moment not one after the other` doubles as a positive timing assertion and a negation of the sequential failure mode. Apply the same pattern to any `signal + spectacle` beat (Moneybags pistol fire + muzzle flash + shell ejection; Dragonwitch flame conjure + ignite; Princess Sweet vine cast + bloom).
- **When the held VFX peak pose is near-canonical, avoid redundant grip transitions at the loop close** (learned 2026-05-05 on General P6 v2). General P6's held planted pose (both hands on pommel, blade in ground) was visually almost identical to the canonical sentinel pose (both hands on pommel, blade hovering). The v2 prompt had a multi-step "left hand drops away → right hand pulls up out of the ground → left hand returns to the pommel" return sequence that looked like a pointless fidget — the user described it as "lifts the hand only to place it back where it already was." **Fix**: when the held peak pose differs from canonical only in **prop position** (not grip), keep both hands locked together on the prop and let the smooth transition happen via prop motion alone — `the blade smoothly rises a small amount out of the ground in the final fraction of the loop with both hands staying locked together on the pommel`. Add explicit ban: `no separate hand lift, no hand release, no two-step grip change at the end of the loop`. **Authoring rule during Stage 3**: compare the held peak pose to canonical. If grip is identical and only prop position differs, structure the return as **prop-motion-only** with grip locked. If grip itself must change (e.g. one-handed peak → two-handed canonical), structure that as a single smooth motion not a multi-step sequence.
- **VFX-heavy concepts drift toward toned-down on regenerations — explicit anti-tone-down language is mandatory** (learned 2026-05-05 on General P6 v3 review). Even with the Architect VFX-intricacy upgrade pattern applied (concrete enumeration, ornate language, comparison anchors), VFX-led concepts will drift toward visibly muted output across regenerations — the third or fourth pass on a "spectacular VFX" concept will read as visibly more restrained than the first. **Fix**: alongside the concrete enumeration, add **explicit anti-tone-down assertion in Constraints** — `the VFX is spectacular and brilliant and elaborate not toned down not restrained not subtle, all elements present together not a single thin generic glow ring not a single basic ring not a single simple effect`. Mirror the assertion in the Negative prompt — `toned down VFX, restrained VFX, subtle VFX, weak impact, soft impact`. **Authoring rule**: every VFX-heavy power concept (P6-class lightburst, Dragonwitch flame, Princess Sweet bloom, Architect diagram bloom) should ship the anti-tone-down assertion in the v1 prompt — don't wait for v3 muting to discover this. Include the explicit visual element list in Constraints so the model has to render all of them rather than collapsing to the simplest subset.
- **Anticipation + weight on the windup needs explicit "heavy / weighty / slow / visible weight" language** (learned 2026-05-05 on General P6 v3 review). VFX-led power moves where the wind-up is supposed to read as heavy and deliberate (Wilhelm-style anticipation applied to non-axe heroes) consistently render as too quick / too light without explicit weight language in the Action body. **Fix**: in Action, write `slowly and heavily lifts`, `slow weighty disciplined arc`, `visible heavy anticipation`, `shoulders coil back as the [weapon] rises`, `chest fills under the [armor]`, `jaw clenches harder`, `the slow lift visibly shows the heavy weight and weighty deliberate effort`. Pair with a **brief charged held-apex beat** before the explosive drop — `holds the apex pose for a brief beat of charged anticipation with the [weapon] glowing brilliantly and his body coiled, then he explosively drives the [weapon] downward`. The held-apex beat is what sells the contrast between slow wind-up and fast strike. **Authoring rule**: any power concept whose impact reads as "heavy / decisive / weighty" in Stage 2 needs the explicit weight language and the held-apex beat in the v1 prompt.
- **Weapon-noun token collision: anatomical part-names that share tokens with a different weapon class cause whole-weapon morph on dynamic motion** (learned 2026-05-05 on Count Wilhelm P2 + P3). Wilhelm's halberd was described in the canonical hero-doc style with `etched silver curved axe-head featuring a top spike`. The word `axe-head` is the correct anatomical term for the chopping head of a halberd, but on **dynamic motion concepts** (P2 horizontal sweep + P3 overhead slam with foreshortening) Seedance read the `axe` token literally and morphed the entire weapon into a **double-bladed battle axe**. Static / front-on concepts (P1 forward thrust, P4 plant, P5 vertical lift, idles) all rendered the halberd correctly. The morph is specific to dynamic motion + foreshortening. **Fix pattern**:
  - **Replace the colliding anatomical term**: `axe-head` → `halberd-blade` / `halberd-head`. Drop standalone `axe` from the positive prompt body.
  - **Lead the Subject with the weapon noun**: move the weapon description **before** character build — `static fixed camera shot of a stocky veteran warlord character with chibi proportions and a broad heavy build, holding a tall ornate single-bladed halberd polearm with [silhouette geometry]...`. Anchoring weapon identity early in the Subject sentence stabilizes the read.
  - **Explicit anti-morph Constraint**: `the weapon is a tall single-bladed halberd polearm with a single curved single-edged silver blade-head and one straight top spike never a double-bladed battle axe never a double-headed axe never a two-bladed axe never a battle axe never a war axe never a twin-bladed weapon, the halberd-head has only one curved blade on the front side of the head it does not have a second blade on the rear side of the head`.
  - **Negative prompt entries**: `double-bladed battle axe, double-headed axe, two-bladed axe, battle axe, war axe, twin-bladed axe head, mirrored blade head, axe with two blades`.
  
  **Authoring rule**: at Stage 1 hero-doc audit, scan every weapon's anatomical-part description for tokens that are themselves recognizable weapon class names — `axe-head` (halberd), `arrow-tip` / `point` (could read as arrow on a spear), `sword-blade` (could read as sword on a polearm), `mace-head` (on a scepter), `dagger-tip` (on a knife-spear), `hammer-head` (on a polearm pickaxe), `spear-tip` / `spear-point` (on a halberd or trident). Replace with weapon-class-neutral terms (`halberd-blade`, `sharp metal tip at the front of the arrow`, `polearm-blade`) or with descriptors that bind the part to the parent weapon (`halberd-head`, `scepter-head`, `polearm-head`). This complements the Architect P4 silhouette-preservation rule — silhouette geometry (A) prevents simplification to a generic shape, but does NOT prevent token-collision morph into a different weapon class. Both fixes need to be applied to dynamic-motion concepts on heroes whose weapon names collide with other classes.
- **Combat-semantic-field collision: non-weapon objects can morph into weapons when surrounded by combat vocabulary** (learned 2026-05-06 on Princess Arrogant P1, generalizes the Wilhelm `axe-head → halberd-blade` lesson to non-weapon objects). Princess Arrogant's war fans were described as `metal folding war fan with sharp pointed steel ribs` and the prompts used `slash`, `slash-trail`, `cut`, `strike` extensively. The first-pass P1 generation rendered an actual sword instead of a fan. The trigger was the dense **combat semantic field** around the fan noun: `war` (combat modifier) + `steel ribs` (weapon-like material noun) + `slash`/`cut`/`blade` (combat verbs) → Seedance collapsed the everyday-object "hand fan" semantic into the weapon class implied by the surrounding vocabulary. **Fix pattern**: re-noun the object to drop combat-semantic modifiers and re-anchor the everyday-object semantic field — `metal folding war fan with sharp pointed steel ribs` → `silver folding hand fan with a sharp pointed metal outer rim`. Drops `war` (combat modifier), drops `ribs` (anatomical-term weapon read), drops `steel` (weapon-y material), uses `outer rim` for shape anchor, keeps the everyday-object framing with `hand fan` and `folding`. Action verbs stay (`slash`, `cut`, `strike`) — the noun anchor does the work. **Authoring rule**: when a hero's weapon is functioning as a stylized everyday object (war fan / decorative cane / ceremonial scepter / inscribed quill / etc.), audit the noun cluster and material/modifier cluster for combat-semantic words (`war`, `battle`, `combat`, `tactical`, `steel`, `iron`, `forged`, `blade`, `ribs`, `edge`, `point`) and replace with civilian-role analogues (`hand`, `decorative`, `silver`, `metal`, `outer rim`, `tip`). Apply during Stage 1 weapon-noun lock — it's harder to repair at Stage 3 because the canonical hero-doc weapon name leaks into prompts. This complements the Architect P2 "re-noun weapons as instruments when the action is non-combat" lesson — that one was about non-combat action verbs around a weapon noun; this one is about combat verbs around a non-weapon noun. Both produce the same morph failure mode in opposite directions.
- **Audio safety rejection extends to fan-VFX / metal-prop spectacle scenes — drop hum / chime / tings entirely** (learned 2026-05-06 on Princess Arrogant P3 + P4, extends Princess Sweet 2026-04-27/28 A6 family 2 music-classifier reject). Princess Arrogant P3 (single-fan boomerang) and P4 (twin boomerang whirlwind) both hit `Output audio has sensitive content` rejection on first-pass. The trigger profile: `cool energy hum` (continuous) + `cool silver pre-launch shimmer hum` (continuous) + `metallic chime` (gold pendant tassels swing) + `sparkle tings` (silver motes) + percussion-like spectacle stack (synchronized snap-catches + impact bursts) + sustained spectacle visuals → music classifier reads as orchestral/musical and rejects. The Princess Sweet incidents centered on body-contact + mood-music words; this incident shifted the trigger family to **metal-prop spectacle with hum/chime/tings vocabulary** — same music-classifier failure mode, different trigger words. **Fix pattern**: drop EVERY `hum` / `chime` / `tings` / `melodic` / `swell` / `harmonic` word from the Sound field. Replace `cool energy hum` → drop entirely or `cool energy crackle`; `metallic chime` → `metallic clack` / `metallic clink` / `metallic snap` / drop entirely; `sparkle tings` → `sparkle bursts` / drop entirely; `pre-launch shimmer hum` → `pre-launch glow` (move from Sound to Action description, no audio counterpart). Always use **percussive material/mechanical sounds** (metallic woosh, fabric creak, silk billow, footstep thud, impact crackle, vortex whoosh, catch-snap) and keep them discrete events not continuous tones. Add explicit `no music no orchestral no chimes no harmonic tones no melodic tones no swells` ban at both ends of the Sound line + positive Constraint asserting only material/mechanical sounds. **Authoring rule**: any hero with a metal-prop weapon (fan, sword, dagger, halberd, hammer, drafting-blade, etc.) producing spectacle-scale VFX should preemptively apply this audio-harden pattern from the v1 prompt — don't wait for the rejection. Also applies to characters with bell-like accessories (chimes, bells, gold pendants) at idle — even idle prompts should drop `chime` and use `clink`/`tinkle` if absolutely necessary, or omit entirely.
- **Horizontal sweep concepts default floaty/weightless without explicit weight language** (learned 2026-05-05 on Count Wilhelm P2). Even with the Architect P4 broad-face-presentation Constraint applied (silhouette preserved), the **timing/inertia** of a horizontal weapon sweep defaults to fast and floaty in Seedance. Wilhelm P2 v1 read as a weightless slash even though every silhouette element was correct. The General P6 v3 weight-language fix (slow/heavy/weighty) generalizes to horizontal sweeps but needs sweep-specific phrasing because the General lesson was about a vertical lift+drop. **Sweep-specific fix pattern**:
  - **Action body** — `slow heavy committed wind-up` + a held anticipation moment + `powerful weighted horizontal arc` driven by `hip rotation + rear leg push-off` + extended `held follow-through pose in a deeply grounded committed stance` + slow heavy return draw. Add a low resonant impact thud SFX at the moment the blade-face passes the centerline of the body to anchor the inertia audibly.
  - **Secondary motion lag/snap language** — cape billows during the slow wind-up and snaps strongly with the body rotation; mustache/hair lag substantially behind the head motion through the sweep showing inertia. Without this, the secondary motion looks too uniform and reads as light.
  - **Positive Constraint** — `strong slow-fast-slow timing contrast that reads as a heavy weapon with real inertia not a floaty weightless slash, the follow-through pose is held in a deeply grounded stance with weight planted on the front leg for a clearly visible extended beat to read weight`.
  - **Negative prompt entries** — `weightless sweep, floaty motion, fast windup, instant strike`.
  
  **Authoring rule**: every horizontal-sweep power concept needs the slow-fast-slow timing language + held grounded follow-through + cape-snap lag language in the v1 prompt. Don't wait for the floaty v1 to discover this.
- **Impact VFX needs enumerated multi-component composition to read as hero-tier, not a single ring + edge crack** (learned 2026-05-05 on Count Wilhelm P3). Wilhelm P3 v1 used a single blue impact ring + brief edge crack on the slam — the user's review was that the impact read as too small/restrained for a heavy halberd slam. This is the same family as the Architect VFX-intricacy upgrade and the General P6 v3 anti-tone-down rule, but for **impact moments specifically**. **Fix pattern — enumerate 4-5 components in the Action body**:
  - (a) **Bright color-named impact flash** at the strike point (`a bright cool blue impact flash at the strike point`)
  - (b) **Multiple expanding shockwave rings** radiating outward from the strike point at ground level (`multiple expanding cool blue shockwave rings radiating outward in flat ground-level bursts from the strike point`)
  - (c) **Scattered energy fragments / motes** bursting outward like shrapnel of light (`scattered cool blue energy fragments and motes bursting outward from the impact like shrapnel of light`)
  - (d) **Short radial cracks of energy** branching outward across the ground from the strike point (`short cool blue radial cracks of energy branching outward across the ground from the strike point`)
  - (e) **Brief fan of lifted energy debris** around the stance (`a brief fan of lifted cool blue energy debris around his stance`)
  
  Pair with **explicit FFLF dissipation Constraint**: `all blue impact VFX fully dissipates and disappears before the loop ends the shockwave rings the energy fragments the radial cracks and the lifted debris all wink out and fade away leaving the ground unmarked` — the impact would otherwise leave permanent ground-marks that break FFLF. Pair with **Sound layering**: a deep loud resonant impact thud + a sharp cool energy crackle + a low bass shockwave rumble + small fizzles as fragments fade. Pair with **Negative prompt entries**: `small impact, weak impact, restrained impact, modest VFX`. **Authoring rule**: every "heavy slam" / "ground-strike" / "decisive impact" power concept needs the 4-5 component enumeration in the v1 Action body. Don't ship a single-ring impact prompt and wait for the user review to upgrade it.

## Technical Specs
- **Input**: Source hero PNG from `Source/Heroes Stylized/`
- **Method**: Image-to-Video (I2V)
- **Models**: Kling 3.0 (API or manual) / Seedance 2.0 (manual)
- **Video format**: `.mov` with alpha channel (transparent background)
- **Audio**: SFX only via ElevenLabs (pipeline WIP) — no voice-overs, no lipsync
- **Animation structure**: Seamless loop — power movement + idle movement (5-7s total)
- **Reference style**: Kingshot / Supercell visual language
